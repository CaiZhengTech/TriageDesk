## Deviations from the brief

1. **Auto-resolve balance (mid-task coordinator correction).** The council amendment said
   "auto-resolve is RARE," which the brief's own case table encoded as all-`escalate`
   placeholders (`"expected_outcome": "REVIEW"` for every row, meant to be manually filled
   in). Mid-task the coordinator clarified that "rare" must not become "never" — an
   all-escalate golden set is a tautology with no negative class for escalation
   precision/recall. I added 4 of the 20 representative cases as `expected_outcome: route`
   (ideal auto-resolve exemplars): ticket 2342 (kept from the auto-selected sample) plus
   3 manual swaps — 12027, 565, 4646 — replacing 3 of the auto-selected Kaggle tickets
   (8196, 9380, 5628 respectively).
   **Why swaps were necessary:** the Kaggle corpus is dominated by verbose synthetic
   enterprise-incident narratives (data breaches, integration failures, campaign metrics)
   with essentially zero genuine "how do I..." tickets — I searched for password-reset,
   VPN, billing-cycle, API-key, and service-status phrasing across the whole `kaggle`
   source and found no clean single-topic informational tickets in the auto-selected 20's
   queues except one (2342, Sales and Pre-Sales). I broadened the search per-queue and
   found 12027 (Dana's existing demo VPN ticket, already in the DB, IT Support), 4646
   (Service Outages/Maintenance, an outage-notification how-to), and 565 (Billing and
   Payments, a plan-pricing/billing-cycle info request) as genuine matches to authored KB
   docs. Each swap is documented with a `notes` field in `golden_expectations.json`
   explaining the substitution and why the current pipeline is still expected to escalate
   it despite the ideal being `route` (near-unreachable margin threshold, per the council
   diagnostic — that gap is what these cases are designed to measure once thresholds are
   re-derived).
2. **Migration import style** adjusted from the raw autogenerate output
   (`from typing import Sequence, Union`) to match the existing
   `73bc987b6ba6_initial_schema.py` convention (`from collections.abc import Sequence`,
   `str | Sequence[str] | None`) — cosmetic only, no behavioral change.
3. **Two trailing comments in `models.py`** (verbatim from the brief) exceeded the
   project's 100-char ruff line limit; shortened/wrapped without changing meaning.
4. **Git branch mishap (self-corrected, not a code deviation).** Mid-task, this shared
   workspace's git branch was found switched to `main` by a concurrent process (a
   "docs: controller operating manual" commit, `8dc1bce`, landed on `main` between my
   `git checkout -b feat/08-golden-set` and my `git commit`), so my commit briefly landed
   on local `main` stacked on top of it. Caught before pushing: moved `feat/08-golden-set`
   to point at that commit (so it now correctly contains my work on top of the latest
   main), reset local `main` back to exactly match `origin/main`, verified with
   `git log`/`git branch -vv`, then pushed only the feature branch. `origin/main` was
   never touched by this.

No deviations from the binding council amendments themselves — the golden set still

## Fix loop 1 (2026-07-12)

Two reviewer findings on PR #32:

- **Critical:** the entitlement-denial trap's auto-assigned ticket id (12044 on the
  original seeding pass) resolved via `customer_ref_for` = `f"customer-{id % 12}"` to
  `customer-8` — an **enterprise**-plan account that IS entitled to `priority_vpn_support`.
  The "trap" was a legitimate grant, not a soft denial. Worse, every reseed minted fresh
  ids (`seed_adversarial_tickets` had no dedup), so which account backed each adversarial
  case was effectively random per run.
- **Important:** seeding was non-idempotent — only `eval_cases` was cleared before
  reseeding; `tickets` rows accumulated as orphans on every run.

### What changed

1. **`triagedesk/evals/adversarial.py`** — each spec now carries a pinned `ticket_id` in
   the 90000+ range (safely above the ~12k-row Kaggle+demo dataset), chosen so
   `id % 12` maps to a documented, fitting account. Comment on each spec states the
   mapping; the trap's comment additionally verifies against `PLAN_ENTITLEMENTS` that the
   resolved account is NOT entitled to the requested feature.
2. **`scripts/build_golden_set.py`** — `seed_adversarial_tickets` now deletes any existing
   `Ticket`/`EvalCase` rows at the pinned ids, then reinserts with an explicit `id=`, before
   returning. Repeated runs converge to identical rows instead of growing new ones.
   Also added the one-line comment reviewer's Minor asked for, next to
   `queue=spec["expected_queue"] or "General Inquiry"`, noting queue is not a graded
   signal for the four `expected_queue=None` adversarial cases in the Task 4 harness.
3. **Dev DB cleanup** — verified the orphaned tickets 12040-12044 were all
   `source='adversarial'` and referenced by zero `runs` rows, deleted them (+ their
   `eval_cases` rows), then re-ran the builder to seed the pinned-id set.
